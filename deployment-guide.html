Skybyte Django App Deployment Guide

First-time Ubuntu droplet setup, GitHub Actions CI/CD, production environment setup, and day-to-day deployment commands.

1. Deployment Overview

This project deploys to a DigitalOcean Ubuntu droplet using Docker Compose. GitHub Actions runs make ci, builds the Docker image, pushes it to Docker Hub, SSHes into the droplet, pulls the image, and runs the app with make prod-pull.

Your droplet SSH command: 
ssh -i ~/.ssh/id_ed25519 root@64.227.147.48

Production traffic goes to nginx, which proxies requests to Gunicorn/Django. The database runs in MySQL inside Docker Compose.

2. Local Project Commands

Command Purpose
make dev-up Start local development stack in detached mode.
make dev-logs Follow local development logs.
make prod-build Build and run the production stack manually from source.
make prod-pull Pull the Docker Hub image and run the production stack on the droplet.
make check Run Django system checks inside Docker.
make check-deploy Run Django deployment checks.

3. First-Time Ubuntu Droplet Setup

3.1 Connect to the droplet

ssh -i ~/.ssh/id_ed25519 root@64.227.147.48

3.2 Update Ubuntu packages

apt update
apt upgrade -y

3.3 Install base tools

apt install -y ca-certificates curl git ufw make unzip

3.4 Install Docker and Docker Compose plugin

This uses Ubuntu packages, which are simple and reliable for a first deployment.

apt install -y docker.io docker-compose-plugin
systemctl enable docker
systemctl start docker
docker --version
docker compose version

3.5 Configure firewall

ufw allow OpenSSH
ufw allow 80/tcp
ufw allow 443/tcp
ufw --force enable
ufw status

3.6 Create deployment folder

mkdir -p /opt/skybyte_fullStack

3.7 Optional: add swap memory

The droplet normally pulls a prebuilt Docker image, so it does less work than a build server. Add swap if the droplet has 1 GB RAM or if MySQL/Docker restarts because of memory pressure.

fallocate -l 2G /swapfile
chmod 600 /swapfile
mkswap /swapfile
swapon /swapfile
echo '/swapfile none swap sw 0 0' >> /etc/fstab
free -h

4. Docker Hub Setup

Create a Docker Hub repository before adding GitHub Secrets.

  1. Open https://hub.docker.com/ and sign in.
  2. Create a repository, for example yourusername/skybytes.
  3. Create an access token from Account Settings -> Security -> Access Tokens.
  4. Use that repository name and token in GitHub Secrets.

5. GitHub Secrets Required

In GitHub, open your repository and go to Settings -> Secrets and variables -> Actions -> New repository secret.

Secret Name Value
DROPLET_HOST 64.227.147.48
DROPLET_USER root
DEPLOY_PATH /opt/skybyte_fullStack
DROPLET_SSH_KEY The full private key content from your local ~/.ssh/id_ed25519 file. Do not use id_ed25519.pub and do not paste the ssh -i ... command.
DROPLET_SSH_PASSPHRASE Only needed if your private key has a passphrase. Leave this secret absent for a key without a passphrase.
PROD_ENV_FILE The full production .env file content.
DOCKERHUB_USERNAME Your Docker Hub username.
DOCKERHUB_TOKEN A Docker Hub access token with permission to push images.
DOCKERHUB_REPOSITORY Your Docker Hub image name, for example yourusername/skybytes.
Important: never commit ~/.ssh/id_ed25519, .env, passwords, or production documents to Git. Put them only in GitHub Secrets or on the droplet.

How to copy DROPLET_SSH_KEY

The value must start with -----BEGIN OPENSSH PRIVATE KEY----- and end with -----END OPENSSH PRIVATE KEY-----.

cat ~/.ssh/id_ed25519

Copy the full output into the GitHub secret named DROPLET_SSH_KEY, including the first and last lines.

7. GitHub Actions CI/CD

The workflow builds a Docker image, pushes it to Docker Hub, then deploys by pulling that image on the droplet. It is located at:

.github/workflows/deploy.yml

It runs automatically when code is pushed to main or master. It can also be started manually from the GitHub Actions tab with Run workflow.

What the workflow does

  1. Runs make ci for Docker Compose config checks, image build, and Django system checks.
  2. Stops if validation fails.
  3. Logs in to Docker Hub with DOCKERHUB_USERNAME and DOCKERHUB_TOKEN.
  4. Builds and pushes DOCKERHUB_REPOSITORY:commit-sha and DOCKERHUB_REPOSITORY:latest.
  5. Uses appleboy/scp-action to upload only docker/, Makefile, and compose.production.yml.
  6. Uses appleboy/ssh-action to write .env, log in to Docker Hub, pull the image, and restart containers.
  7. Runs make prod-pull and make status on the droplet.

Production compose file

The droplet uses compose.production.yml. It requires WEB_IMAGE, which the workflow writes into .env automatically.

8. First Deployment

  1. Commit and push the workflow file, Docker files, compose files, Makefile, and deployment guide.
  2. Add all GitHub Secrets listed above.
  3. Open GitHub Actions in your repository.
  4. Select Build and Deploy.
  5. Click Run workflow, or push to main/master.
  6. Wait for the CI, Docker Hub push, and droplet deploy jobs to finish.
  7. Open http://64.227.147.48.

9. Server Commands After Deployment

SSH into the droplet:

ssh -i ~/.ssh/id_ed25519 root@64.227.147.48

Go to the deployment folder:

cd /opt/skybyte_fullStack
Command Purpose
make status Check running containers.
make prod-logs Follow all production logs.
docker compose -f compose.production.yml logs -f web Watch Django/Gunicorn logs.
docker compose -f compose.production.yml logs -f nginx Watch nginx logs.
docker compose -f compose.production.yml exec web python manage.py migrate Run migrations manually.
docker compose -f compose.production.yml exec web python manage.py createsuperuser Create a Django admin user.
docker compose -f compose.production.yml restart web Restart only Django/Gunicorn.

Manual image redeploy or rollback

GitHub Actions writes WEB_IMAGE into .env. To manually roll back, edit WEB_IMAGE to an older Docker Hub tag, then restart the production stack.

cd /opt/skybyte_fullStack
nano .env
make prod-pull
make status

10. Domain and HTTPS

Point your DNS records to the droplet:

Type Name Value
A @ 64.227.147.48

After HTTPS is configured, update PROD_ENV_FILE:

DJANGO_SECURE_SSL_REDIRECT=true
DJANGO_SECURE_HSTS_SECONDS=31536000
DJANGO_SECURE_HSTS_INCLUDE_SUBDOMAINS=true

Only set DJANGO_SECURE_HSTS_PRELOAD=true after you understand browser preload requirements and are sure every subdomain supports HTTPS.

11. Troubleshooting

Port 80 does not open

ufw status
make status
docker compose -f compose.production.yml logs nginx --tail=100

Django shows bad request

Add the IP or domain to DJANGO_ALLOWED_HOSTS in the PROD_ENV_FILE secret and redeploy.

Database login fails

If MySQL was already initialized with an older password, changing DB_PASSWORD will not reset the existing database volume. Use the original password or intentionally recreate the database volume after backing up data.

Docker Hub push fails

  • Check DOCKERHUB_USERNAME.
  • Check DOCKERHUB_TOKEN.
  • Check DOCKERHUB_REPOSITORY, for example yourusername/skybytes.

Droplet cannot pull image

  • Make sure Docker is installed on the droplet.
  • Make sure the workflow logs in to Docker Hub successfully.
  • If the Docker Hub repository is private, confirm the token has pull access.

CI build fails because of memory

The image builds on GitHub Actions, not on the droplet. If Docker services on the droplet restart because of memory pressure, add swap using the command in section 3.7.

GitHub Action cannot SSH

  • Make sure DROPLET_HOST is 64.227.147.48.
  • Make sure DROPLET_USER is root.
  • Make sure DROPLET_SSH_KEY contains the full private key, including the first and last lines.
  • If the log says ssh.ParsePrivateKey: ssh: no key found, the secret is usually a public key, an SSH command, empty text, or a key copied without the begin/end lines.
  • If your key has a passphrase, add DROPLET_SSH_PASSPHRASE as a GitHub secret.
  • Confirm you can connect locally with ssh -i ~/.ssh/id_ed25519 root@64.227.147.48.

12. Final Safety Checklist

  • .env is ignored by Git and not committed.
  • Private SSH keys are stored only in GitHub Secrets and on your local machine.
  • DEBUG=false in production.
  • DJANGO_SECRET_KEY is long, random, and private.
  • DB_PASSWORD is strong and private.
  • Firewall allows only SSH, HTTP, and HTTPS.
  • Backups are configured before real customer data goes live.
